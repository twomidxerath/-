# 데이터베이스

## 용어 정리

Relation : 데이터가 저장되는 2차원 테이블 전체를 말한다.

Attribute : 속성. 표의 열을 의미하고, 데이터의 특징을 나타낸다. 학번, 이름, 학과 등이 이곳에 속한다.

Tuple : 표의 행으로, 실제 데이터 한 건을 의미한다. 위에서 나온 Relation이 이 Tuple의 집합으로, Tuple 사이의 순서는 그다지 중요하지 않다.

Domain : 하나의 Attribute가 가질 수 있는 값의 범위. 학년 속성의 도메인은 1~4가 될 것이다.

Schema : 데이터베이스의 설계도. 자주 변경되지 않는다.

Instance : 특정 시점에 저장된 실제 데이터를 의미한다. 수시로 데이터가 업데이트되며 변화한다.


## 관계형 데이터베이스를 사용하는 이유

가장 큰 이유는 데이터 독립성 때문이다. 기존의 파일 시스템은, 데이터에 새로운 컬럼을 추가하거나 구조가 바뀌면 코드를 전부 뜯어고쳐야만 한다.하지만, DBMS(Database Management System)를 사용하면, 데이터의 구조와 응용 프로그램이 철저하게 분리되어 데이터의 구조가 변화해도 프로그램은 영향받지 않는다.

## SQL

SQL은 기존에 배웠던 프로그래밍 언어와는 구조적으로 큰 차이가 있다. 기존의 프로그래밍 언어들은 별도의 라이브러리 없이는 데이터를 조회하기 위해선 반복문과 조건문을 사용해 직접 데이터를 순회하며 검색해야 한다. 하지만, SQL은 DBMS와 응용 프로그램이 철저하게 분리되어있기 때문에 이것이 불가능하다. 대신, 내가 어떤 데이터를 찾는지 조건만 데이터베이스에 입력하면 알아서 DBMS가 데이터를 가져온다.

SQL 명령어는 목적에 따라 크게 3가지로 나뉜다.

### 1. DDL(데이터 정의어, Data Definition Language)

schema 자체를 설계하고 변경할 때 사용한다. 다른 프로그래밍 언어로 치면 선언부에 속한다.

CREATE : 테이블을 새로 만든다.

ALTER : 테이블 구조를 변경한다.

DROP : 테이블 구조 자체를 완전히 날려버린다.

### 2. DML(데이터 조작어, Data Manipulation Language)

실제 instance 값을 변경할 때 사용한다. 제일 많이 사용해야 하는 코드다.

INSERT : 테이블에 새로운 튜플 추가.

SELECT : 원하는 데이터를 검색.

UPDATE, DELETE : 데이터를 수정하고 삭제.

### 3. DCL(데이터 제어어, Data Control Language)

데이터베이스 접근 권한을 관리한다.

GRANT, REVOKE : 각각 사용자에게 권한을 부여, 회수.



## DDL

### CREATE

표를 생성하는 명령어로, C의 구조체 선언과 상당히 유사한 구조를 띈다.

```sql

CREATE TABLE 테이블이름 (
    컬럼명1 데이터타입 [제약조건],
    컬럼명2 데이터타입 [제약조건],
    ...
);
```

테이블 이름과 컬럼명은 사용자가 직접 설정하는 것이고, 우리가 생각해야 할 것은 딱 두 가지이다. 데이터타입과 제약조건.


1. 데이터타입

다른 프로그래밍 언어의 데이터 타입과 겹치거나 비슷한 부분이 있다. 자주 사용되는 표준 타입들은 다음과 같다.

INT : 정수형 데이터

VARCHAR(n) : 최대 n의 길이를 가지는, 가변길이 문자열.

CHAR(n) : 무조건 n바이트의 공간을 차지하는 고정 길이 문자열. 학번/주민등록번호같이 반드시 길이가 일정해야 하는 데이터에 사용된다.

DATE/DATETIME : 날짜와 시간.


2. 제약조건

테이블에 쓰레기 데이터가 들어오는 것을 방지하기 위해 컬럼별로 다른 제약조건을 거는 것.

PRIMARY KEY : 제일 중요한 제약조건이다. 테이블에서 각 튜플을 유일하게 식별할 수 있는 대표 컬럼이라고 볼 수 있다. 중복된 값도 들어올 수 없고, 값을 비워두는 것도 불가능하다. 무조건 고유한 값이 이 안에 들어와있어야 한다. 학번, 주민번호 등 식별자에 이러한 조건을 걸어두는 편이다.

NOT NULL : 값을 비워두는 것만 금지한다.

UNIQUE : 중복되는 값이 들어오는 것만 금지한다. 값을 비워두는 것은 허용한다.


### ALTER

이미 만들어진 테이블을 수정할 때 사용한다. 크게 세 가지 기능을 가진다.

1. 새로운 열 추가하기 (ADD)

```sql
ALTER TABLE Student ADD 전화번호 VARCHAR(20);
```

2. 기존 열의 데이터 타입 변경하기 (MODIFY)

```sql
ALTER TABLE Course MODIFY 과목코드 VARCHAR(10);
```

이때 바꾸기 전의 데이터 타입은 중요하지 않으니, 바꿀 데이터 타입만 명시하면 된다. DBMS 종류에 따라 MODIFY 대신 ALTER COLUMN을 사용하기도 한다.

3. 기존 열 삭제하기 (DROP COLUMN)

```sql
ALTER TABLE Student DROP COLUMN 전화번호;
```

이때 전화번호 열에 들어있던 데이터가 전부 지워진다. 다른 데이터는 건드리지 않는다.


### DROP

테이블의 schema와 그 안에 든 모든 데이터를 통째로 삭제하는 명령어이다. 복구하기 까다로우니 신중하게 사용해야만 한다.

```sql
DROP TABLE Course;
```



## DML

백엔드 코드를 짤 때 제일 많이 작성해야 하는 부분이다. 이를 CRUD (Create, Read, Update, Delete) 작업 이라고 부르기도 한다.

### INSERT

테이블에 새로운 튜플을 추가할 때 사용한다.

```sql
INSERT INTO Student (학번, 이름, 전공)
VALUES (101, '홍길동', '소프트웨어학과');
```

데이터를 추가할 때는, PRIMARY KEY나 NOT NULL로 지정되어 반드시 값을 넣어야 하는 속성이 아니라면, 값을 넣지 않아도 된다. 자동으로 NULL 값이 들어갈 것이다.

참고로, SQL은 줄바꿈과 띄어쓰기를 전혀 신경쓰지 않는 방식을 채용했다. 위의 코드는 2줄이지만, 1줄로 쭉 늘여쓴 것과 완전히 동일한 효과를 낸다.


### SELECT

SQL에서 가장 중요한 문법이라고 해도 과언이 아닌 문법이다. 이 명령어를 어떻게 다루는 지에 따라 데이터베이스의 최적화 정도가 천차만별로 갈린다. 관계 대수의 셀렉션과 프로젝션이 합쳐진 문법이다.

```sql
SELECT 이름 FROM Student WHERE 전공 = '소프트웨어학과';
-- 학생 테이블 전체에서 전공이 소프트웨어학과인 모든 데이터를 조회한다.

SELECT * FROM Student;
-- 이렇게 별다른 조건을 걸지 않고 학생 테이블 전체를 조회하는 것도 가능하다.
```


### UPDATE

이미 들어간 데이터를 다른 값으로 변경할 때 사용한다. 

```sql
UPDATE Student SET 전공 = '인공지능' WHERE 학번 = 102;
-- 학번이 102인 학생을 찾아 전공을 인공지능으로 바꾸는 코드.
```


### DELETE

테이블에서 특정 튜플을 아예 지워버릴 때 사용한다.

```sql
DELETE FROM Student WHERE 학번 = 101;
-- 학번이 101인 학생을 테이블에서 삭제.
```


### ORDER BY

가져온 데이터를 오름차순이나 내림차순으로 정렬할 때 사용. 별다른 정렬 알고리즘을 짤 필요는 X

```sql
SELECT * FROM Student ORDER BY 학번 DESC;
-- DESC는 내림차순, ASC는 오름차순으로 정렬하라는 뜻.
```

만약 값이 같은 값끼리 다른 컬럼을 사용해 또다시 우선순위를 부여하고 싶은 경우, , 를 사용해 뒤에 추가적인 정렬 기준을 붙이면 된다.

```sql
SELECT ID, LENGTH 
FROM FISH_INFO
ORDER BY LENGTH DESC, ID ASC
LIMIT 10;
-- 프로그래머스 문제 발췌. 생선을 길이로 내림차순 정렬하고, 값이 같은 것들끼리는 id로 오름차순 정렬하는 코드. 맨 뒤에 붙은 LIMIT 10은 10까지 보여주라는 의미이다.
```

### 데이터 통계 내기

COUNT, SUM, AVG, MAX, MIN 등이 존재한다. 사용법은 해당 명령어 옆에 ()를 치고 안에 통계를 낼 컬럼을 입력하면 된다. *로 모든 컬럼을 지정하는 것도 가능하다.

```sql
SELECT COUNT(*) FROM Student;
-- 학교에 등록된 전체 학생 수를 센다.
```

### GROUP BY

전체 통계가 아니라 특정 그룹별로 통계를 내고 싶을 때 사용한다.

```sql
SELECT 전공, COUNT(*) 
FROM Student 
GROUP BY 전공;
-- 컴퓨터가 먼저 전공이 같은 학생들끼리 임시로 그룹을 지은 후, 각 그룹에 속한 학생 수를 계산해서 보여준다.
```


### HAVING

GROUP BY로 묶은 그룹들에 대해 조건을 걸고 싶을 때 사용한다. WHERE은 개별 튜플의 조건을 확인하고, 그룹의 조건을 확인할 때는 반드시 HAVING을 사용해야만 한다.

```sql
SELECT 전공, COUNT(*) 
FROM Student 
GROUP BY 전공 
HAVING COUNT(*) >= 2;
-- 위의 코드와 비슷하지만, HAVING을 사용해 학생 수가 2명 이상인 전공만 필터링해서 학생 수를 출력한다.
```

SQL의 모든 문법에서 적용되는 주의사항인데, WHERE은 반드시 주의해서 사용해야만 한다. SQL은 일반 프로그래밍 언어와 달리 디스크에 영구적으로 저장된 데이터베이스에 접근하기 때문에 이러한 실수가 훨씬 치명적으로 다가오기 때문이다.



SQL의 논리적 실행순서는 다음과 같다. 코드 순서가 아니다. 이 순서를 바탕으로 아무리 복잡한 쿼리라도 실행 결과를 예측해볼 수 있다.

FROM : 어떤 테이블에서 가져올지 결정.

WHERE : 어떤 행을 골라낼지 조건을 결정. (데이터 골라내기 성공)

GROUP BY : 골라낸 데이터를 어떻게 묶을지를 결정.

HAVING : 묶인 그룹들 중 어떤 것을 남길지를 결정. (이제 데이터 조작은 끝.)

SELECT : 어떤 열을 출력할지를 결정.

ORDER BY : 결정된 데이터들을, 어떤 순서로 출력할지 결정.


### JOIN

JOIN에는 크게 네 종류가 있다.

INNER JOIN : 두 테이블 모두에 공통으로 존재하는 데이터만 교집합처럼 가져옴.

LEFT OUTER JOIN : 왼쪽 테이블은 무조건 가져오고, 오른쪽 테이블에서 조건이 맞는 것만 붙여준다. 짝이 없는 빈칸은 NULL처리한다.

RIGHT OUTER JOIN : LEFT OUTER JOIN의 반대다.

FULL OUTER JOIN : 양쪽 테이블의 모든 데이터를 합집합처럼 전부 살린다.


```sql
SELECT Student.이름, Enrollment.과목코드
FROM Student
INNER JOIN Enrollment
ON Student.학번 = Enrollment.학번;

-- Student 테이블과 Enrollment 테이블을 합치되, ON 뒤에 붙은 조건처럼 학번이 같은 데이터끼리만 짝을 지어서 이름과 과목코드를 출력한다.
```

위 코드는 INNER JOIN의 코드지만, 다른 JOIN 방식도 사용 방식은 동일하다. 출력하는 결과가 다를 뿐.

참고로, MYSQL에서는 FULL OUTER JOIN은 지원하지 않기 때문에, LEFT OUTER JOIN, RIGHT OUTER JOIN을 각각 사용한 후 두 결과를 합쳐야 한다.


## 서브쿼리

쿼리 안에 또다른 쿼리가 위치하는 것을 서브쿼리라고 부른다. 다른 언어에서, 함수의 결괏값을 다른 함수의 인자로 집어넣는 것과 비슷하다고 볼 수 있다. JOIN만으로 해결하기 까다로운 복잡한 조건들도 처리할 수 있는 것이 장점이다.

1. 중첩 서브쿼리

WHERE 절에는 집계 함수, COUNT나 AVG, SUM 등을 사용할 수 없다. 때문에 학교 평균 평점보다 높은 학생들을 찾기 위해서는, 서브쿼리가 필수적이다.

```sql
SELECT 이름, GPA
FROM Student
WHERE GPA > (
    SELECT AVG(GPA)   -- 이 서브쿼리가 먼저 실행되어서 '3.6' 같은 단일 숫자로 변환돼!
    FROM Student
);
```

2. 인라인 뷰

FROM절에 들어가는 서브쿼리이다. 이 방식은 서브쿼리의 결과를 마칯 임시 테이블처럼 취급하는 방식이다. 

```sql
SELECT 임시테이블.이름, 임시테이블.GPA
FROM (
    -- 소프트웨어학과 학생만 걸러낸 가상 테이블(인라인 뷰) 생성
    SELECT 학번, 이름, GPA 
    FROM Student 
    WHERE 전공 = '소프트웨어학과'
) AS 임시테이블   -- 이 임시 테이블에 반드시 별명(Alias)을 붙여줘야 해!
WHERE 임시테이블.GPA >= 4.0;
```


3. 스칼라 서브쿼리

SELECT절에 들어가는 쿼리이다. 

```sql
SELECT 
    이름,
    (
        -- 각 학생(바깥쪽 쿼리의 학번)마다 수강 중인 과목 개수를 세는 쿼리
        SELECT COUNT(*) 
        FROM Enrollment 
        WHERE Enrollment.학번 = Student.학번
    ) AS 수강과목수
FROM Student;
```


## 데이터 모델링

데이터 모델링이란, 현실 세계의 복잡한 정보들을 컴퓨터가 이해할 수 있는 논리적인 구조, 테이블로 변환하는 과정이다. 다음과 같은 3단계의 과정을 거친다.

1. 개념적 모델링(Conceptual) : 핵심 개체(Entity)와 그들 사이의 관계만을 정리.

2. 논리적 모델링(Logical) : 구체적인 Attribute와 Primary Key 등을 정의하며, 특정 데이터 모델링에 맞게 설계.

3. 물리적 모델링(Physical) : 실제 어떤 DBMS를 쓸지, 인덱스는 어떻게 걸지 결정.

당연하게도 데이터 모델링에도 여러 가지 방식이 존재한다. 가장 널리 쓰이는 모델링은 ER Model로, 개체-관계 모델이다. ERD(Entity-Relationship Diagram)으로 표현한다. ERD는 세 가지의 요소로 구성된다.

Entity : 저장할 가치가 있는 중요 정보의 단위. 직사각형으로 그린다.

Attribute : 개체가 가지는 세부 정보들. 보통 타원형으로 그린다.

Relationship : 개체와 개체 사이의 연관성. 보통 마름모로 표현한다.

관계에는 여러 종류가 있다. 이는 모델링에서 상당히 중요한 부분이며, Cardinality라고 부른다.

1:1(일대일) 한 개체가 다른 개체와 일대일로 연결된다. 국민-주민등록번호 같은 케이스가 여기에 해당한다.

1:N(일대다) 한쪽은 하나인데, 다른 쪽이 여럿일때. 학과-학생이 여기에 해당한다.

N:M(다대다) 양쪽 모두 여러 개와 연결될 수 있는 경우. 학생-수강과목 같은 케이스가 여기에 해당한다.


## 정규화

데이터 모델링에서 생긴 데이터의 중복을 제거하고, 구조를 깔끔하게 쪼개는 과정을 정규화라고 부른다.

정규화를 하지 않고 거대한 하나의 테이블에 모든 데이터를 때려박으면, 다음과 같은 이상현상들이 발생한다.

삽입 이상 (Insertion Anolmaly) : 새 데이터를 넣고 싶은데, 원치 않는 정보까지 억지로 넣어야 하는 상황

갱신 이상 (Update Anomaly) : 중복된 데이터 중 일부만 수정되어 데이터 간 불일치가 발생하는 상황

삭제 이상 (Deletion Anomaly) : 정보를 삭제할 때 엉뚱한 정보가 같이 삭제되는 상황

이러한 이상들을 해결하기 위해 정규화를 하는데, 일반적으로는 3정규화, 3NF까지만 진행한다.

제 1정규화 (1NF) : Atomic Value. 모든 컬럼이 단 하나의 값만 가지게 정규화한다. 한 칸에 여러 개의 정보가 들어있거나, 리스트가 들어있는 경우 이를 별도의 행으로 분리한다.

제 2정규화 (2NF) : 부분 함수 종속 제거. 기본키가 여러 컬럼의 조합(복합키)이거나, 기본키의 일부분에 의존하는 컬럼이 있어서는 안 된다.

제 3정규화 (3NF) : 이행적 함수 종속 제거. 기본키가 아닌 컬럼들끼리 서로 의존하는 것을 제거한다. 

1정규화는 직관적이고 간단하지만, 나머지 둘은 그렇지 않다. 2정규화는, 기본키가 두 개 이상의 컬럼으로 합쳐진 복합키 일 때 고려해야 할 부분이다. 학번, 과목코드를 가진 수강성적이라는 테이블이 존재한다고 가정하자. 학생의 이름을 아는 데는 학번이면 충분하고, 무슨 과목인지 알기 위해서는 과목코드로 충분하다. 하지만, 학생의 성적을 조회하기 위해서는 두  기본키가 모두 필요하다. 제 2정규화는, 이러한 테이블을 학번만 기본 키로 쓰는 테이블, 과목코드만 기본 키로 쓰는 테이블, 둘 모두가 필요한 테이블로 각각 분리하는 과정이다.

3정규화는 이행적 함수 종속 제거이다. 다음과 같은 학생 테이블이 있다라고 가정하자.

학번을 기본 키로 가지고, 이름, 지도교수, 지도교수실이 컬럼으로 들어가 있다. 여기서 지도교수실은, 물론 학번을 알면 지도교수를 알기에 알 수 있기는 하지만, 학번->지도교수->지도교수실이라는 종속 관계가 존재한다. 지도교수가 변경되면 지도교수실도 같이 변경되어야 한다라는 의미이다. 반대로, 교수실이 바뀌었을 때 모든 학생의 데이터를 수정해야 하는 수고로움이 발생한다. 이를 갱신 이상이라고 부른다.

이 경우, 테이블을 다음과 같이 둘로 쪼갠다. 학생 테이블에는 학번, 이름, 지도교수를 넣고, 따로 지도교수 테이블을 만들어 지도교수, 교수실을 만들면 이러한 문제가 해결된다.


## Transaction

트랜잭션이란, 운영체제의 Lock과도 아주 밀접한 관련이 있다. 은행 시스템을 생각해보자. 누군가 만 원을 자신의 계좌에서 타인의 계좌로 이체하려고 한다. 이때, 계좌에서 만원이 빠져나갔는데 오류로 인해 만 원이 입금되지 않았다면? 만 원이 허공에 증발한 셈이 된다. 이러한 문제를 방지하기 위한 개념이 트랜잭션이다.

트랜잭션에는 4가지의 규칙이 존재한다. 이를 초성만 다 따서 부르면 ACID가 된다.

원자성(Atomicity) : 트랜잭션 내의 연산은 전부 성공하든 전부 실패하든 원자적으로 진행되어야 한다.

일관성(Consistency) : 트랜잭션이 끝난 후에도 데이터베이스는 미리 정해진 규칙을 모두 만족해야 한다.

격리성(Isolation) : 여러 트랜잭션이 동시에 실행될 때, 서로의 연산 과정에 개입할 수 없다.

지속성(Durability) : 성공적으로 완료된 트랜잭션의 결과는 시스템이 고장나더라도 영구적으로 기록되어야 한다.

실제로 SQL을 짤 때, 트랜잭션을 확정짓거나 취소할때는 COMMIT(깃허브의 그거 맞다. 변경사항을 저장하고 DB에 반영하는 명령어이다.), ROLLBACK(트랜잭션 시작 전의 상태로 되돌리는 것.)

실제로 트랜잭션이 일어날 때는, 내부적으로 복잡한 상태 변화가 일어난다. 단순히 커밋, 롤백으로만 나뉘지 않는다.

Active : 트랜잭션이 시작되어 명령어가 실행 중인 상태.

Partially Committed : 명령이 전부 실행되었지만, 아직 DB에 반영되지는 않은 상태.

Committed : 데이터 처리가 성공하여 DB에 영구적으로 반영된 상태.

Failed : 오류가 발생.

Aborted : 실패 후 트랜잭션 시작 전으로 데이터를 전부 롤백시킨 상태.

## 동시성 제어

DB에서는 동시성을 Lock으로 컨트롤한다. Lock 의 안정성을 위해서 2단계 잠금 규약을 사용하는데, 트랜잭션이 데이터를 사용하기 전에 무조건 Lock을 걸고, 끝난 뒤에만 해제하는 규약이다. 크게 확장 단계와 수축 단계로 나뉘는데, 확장 단계는 필요한 모든 Lock을 획득하고, 수축 단계는 획득한 Lock을 순차적으로 해제하는 단계이다.

DB에서 Deadlock이 발생하면, DB 엔진은 한쪽 트랜잭션을 강제로 죽이는 방식을 택한다.

MVCC(Multi_version Concurrency Control)

최신 데이터베이스가 성능을 위해 사용하는 핵심 기술. 데이터를 수정할 때, 이전 버전의 데이터를 따로 남겨둠으로써 락에 상관없이 읽기와 쓰기가 가능하다.



## 인덱스

인덱스는 방대한 데이터 안에서 원하는 정보를 빠르게 찾아내기 위한 색인과 같다. 데이터베이스를 처음부터 끝까지 무식하게 검색하려면 자료 하나 찾는데 엄청난 양의 시간을 소모해야 한다.

인덱스는 B-Tree라는 자료구조를 사용한다. B-Tree는 루트 노드 하나에 수많은 자식 노드가 달릴 수 있는 트리로, 다음과 같은 이유 때문에 데이터베이스에서 채용하게 되었다.

1. 균형 잡힌 높이 : 어떤 데이터를 찾더라도 루트 노드에서 리프 노드까지 가는 거리가 비슷하다.
2. 디스크 I/O 최적화 : 한 노드에 여러 개의 데이터를 담을 수 있어 트리의 높이를 매우 낮게 유지하는 것이 가능하다. 
3. 탐색 성능 : 데이터 검색 시 시간 복잡도가 O(log n)으로 효율적이다.


인덱스는 크게 두 종류로 나뉜다. 클러스터형 인덱스, 비클러스터형 인덱스.

클러스터형 인덱스 : 사전과 같다. 실제 데이터 행들이 인덱스 순서(기본키 순서)대로 정렬되며, 기본 키 기반으로 작동하기 때문에 테이블당 1개만 생성 가능하다. 조회는 아주 빠르지만 삽입/수정이 느리다.

비클러스터형 인덱스 : 별도의 인덱스 페이지를 만들고 실제 주소를 참조한다. 테이블 하나에도 여러 개 생성이 가능하며, 조회 속도는 클러스터형보다 느리지만 삽입/수정 측면에서 더 유연하다.





## JSON

기존의 SQL 테이블은 사전에 정의된 타입만을 허용하고, 모든 행이 같은 컬럼을 가져야 하는 등 상당히 까다로운 제약 조건이 붙어있었다. 이는 무결성을 지키는 데에는 최고의 선택이지만, 업데이트하고 데이터를 삭제하는 데에는 상당한 시간이 소모되는  결과로 돌아왔다. 이러한 단점을 해결하기 위한 것이 JSON이다.

JSON은 고정된 틀 없이 해시 테이블 구조를 사용해 Key : Value 쌍으로 데이터를 저장한다. 따라서 데이터 값이 달라도 유연하게 데이터를 저장하고 추가하고 삭제하는 것이 가능하다. 또한, 객체 안에 또다른 객체를 넣거나 배열을 추가할 수도 있다.

물론 JSON이 장점만을 가지는 것은 아니다. 결국 Key를 통해 데이터를 찾기에 데이터를 읽을 때마다 문자열 파싱이 필요하고, JSOn은 데이터 형을 구분하지 않고 자료를 저장하기에 인덱싱에서 어려움을 겪는다.

JSON은 기존의 SQL 테이블과는 다르게, 데이터 통합시에도 정규화할 필요 없이 그저 데이터를 추가하기만 하면 되고, 객체 안에 넣기만 하면 된다. 또한, 복잡한 JOIN 과정도 필요없이 하나의 컬럼 안에 직관적으로 데이터를 넣어 다룰 수 있다.

물론 JSON이 장점만 가지지는 않기에, 기존의 SQL 테이블과 섞어 하이브리드 식으로 사용하기도 한다. 무결성이 더 중요한 정보들은 테이블로 관리하고, 자주 변경되는 정보들은 JSON으로 관리하는 식. 


## Not Only SQL(NoSQL)

전통적인 관계형 데이터베이스(RDB)의 한계를 극복하기 위한 새로운 흐름. SQL을 완전히 사용하지 않는 것이 아니라, 필요에 따라 여기저기서 다양한 저장 방식을 활용한다라는 의미. 크게 4가지 방향이 있다.

1. 문서지향(Document-oriented) DB : JSON, BSON과 같은 반정형 문서 형태로 데이터를 저장. 개발이 아주 직관적이다.

2. 키-값 DB : 키 하나에 값 하나 매핑. 읽고 쓰는 속도가 아주 빠르고, 캐싱/세션 관리에 사용.

3. 와이드 컬럼(Column-family) DB : 행마다 다른 개수의 컬럼을 가질 수 있는 구조. 큰 데이터를 다루는 것에 강하며, 빅데이터 처리에 사용한다.

4. 그래프 DB : 그래프 구조를 사용해 데이터간의 관계를 표현한다. 소셜 네트워크나 추천 시스템같이 복잡하게 얽힌 관계를 탐색하는 데에 사용된다. 


NoSQL은 다음과 같은 장점들을 가진다.

1.Schema-less : 데이터를 넣기 전에 Schema, 즉 구조를 미리 정의할 필요가 없다. 컬럼 하나에 다양한 데이터가 들어갈 수 있어 유연하다.
2. 수평적 확장성 (Horizontal Scalability) : 비싼 서버 하나 대신, 저렴한 서버 여러 개를 연결해 사용하는 방식에 강하다.
3. 비정규화 (Denormalization) : JOIN 연산의 비용을 줄이기 위해 일부러 데이터를 여러 테이블에 중복 저장한다. 이러면 JOIN할 필요가 없이 데이터 조회 속도가 비약적으로 상승한다.
